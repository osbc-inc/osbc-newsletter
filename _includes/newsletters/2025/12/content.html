OSBC LETTER
2025년 12월
React2Shell 취약점
오픈소스 보안 관리 관점에서의 대응 방향
최근 React 서버 컴포넌트(Server Components) 환경에서 사용되는 일부 하위 라이브러리와 관련하여, 원격 코드 실행(RCE)로 이어질 수 있는 매우 심각한 React2Shell 취약점 이슈가 확인되었습니다.

해당 취약점은 React 서버 컴포넌트 생태계에서 사용되는 구현체 중 일부 입력 처리 흐름에서 발생하며, 특정 조건에서는 의도하지 않은 코드 실행으로 이어질 수 있는 위험을 포함하고 있습니다.

본 이슈는 Snyk Security Advisory를 통해 공개되었으며, CVE-2025-55182로 식별되고 있습니다.

이번 취약점은 React 서버 컴포넌트 환경에서 사용되는 react-server-dom-webpack, react-server-dom-parcel, react-server-dom-turbo 등 하위 구현체와 연관되어 있으며, 패치 또는 업데이트가 제공되는 경우 공식 권고에 따라 신속히 적용하는 것이 권장됩니다.

이에 OSBC는 해당 취약점을 기준으로, 당사의 주요 SCA 도구들이 어떤 기준과 단위로 취약점을 식별·관리하는지를 비교 확인하기 위한 테스트를 진행했습니다.
도구별 React2Shell 탐지 결과
Snyk – 하위 라이브러리 및 버전 단위 식별
Snyk은 React 서버 컴포넌트 환경에서 사용되는 하위 라이브러리와 버전 정보를 기준으로 취약점을 식별함으로써, 실제 취약점이 발생하는 구성 요소를 비교적 명확하게 확인할 수 있었습니다.
FossID – React 제품 단위 기준 관리
FossID는 React를 제품 단위로 인식하여 취약점을 관리하며, CVE/CPE 기반의 표준화된 취약점 관리 관점을 유지하고 있음을 확인할 수 있었습니다.
Insignary Clarity – 프로젝트 구성 요소 중심 식별
Insignary Clarity는 React 기반 프로젝트의 구성 요소 전반을 기준으로, 프로젝트 관점에서의 취약점 노출 현황을 파악하는 데 유용했습니다.
React2Shell 대응, 왜 관리 관점이 중요한가
이번 React2Shell 사례에서 확인할 수 있듯이, 구조가 복잡한 오픈소스 환경에서는 단순히 취약점의 존재 여부만으로 실제 운영 리스크를 판단하기 어렵습니다.

동일한 CVE라 하더라도, 하위 라이브러리 기준, 제품 단위 기준, 프로젝트 구성 기준에 따라 도구별 탐지 결과와 대응 범위는 서로 다르게 나타날 수 있습니다.

OSBC는 이러한 오픈소스 보안 환경의 특성을 고려하여, 조직의 개발 구조와 운영 환경에 적합한 오픈소스 구성 관리 및 취약점 대응 체계 수립을 지원하고 있습니다.
오에스비씨(OSBC)는 국내 최초, 최다 오픈소스 컨설팅 및 솔루션 공급 이력을 기반으로한 선도적인 오픈소스 전문 기업입니다.
